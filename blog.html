Writing tools

I asked content designer Emma Cohen what tools she uses day to day. Her copy has to be pitched at the right level for her audience.

My immediate response was that I don't use many tools - I use my good ol' brain and eye for detail mostly. But I do use some tools, depending on the situation.

  1. WebFX - in my current role, I use the WebFX readability tool quite a bit: https://www.webfx.com/tools/read-able/ - Our team is rewriting a lot of complex content at scale and we have a goal of hitting a Flesch-Kincaid score of at least 70.   4. Not sure if this is classed as a tool, but I use Google Trends quite a bit if I'm researching language that users use (especially if I don't have access to any user research) - https://trends.google.com/trends/.

Traditional document editors: Word, Google Docs

The web is an interactive medium but it contains static documents. Traditional writing and publishing still exists on the web because older technologies never quite die out.

Content designers may still use established tools like MS Office, Google documents or Confluence to write copy. A large part of content design is about collaboration: they find the right people in an organisation to write and review web content. There's often a 'review cycle' or sign-off process to make sure the copy is legal and compliant.

For documents like a Terms and conditions page or a help article Word and Google docs tools are likely to do the job. Allowing people on the project to comment and make suggestions on the copy is also useful on a lot of projects.

'No code' solutions

I've seen at least one designer advertise their approach as 'no code'. Design platforms like Canva and Figma let people develop layouts and write content without using any HTML or CSS, the languages of the web. Webflow goes slightly further and lets you develop and host interactive web services with no code.

If the project is using a CMS like Wordpress or Umbraco, or a web hosting solution like Wix or Shopify, designers can also use WYSIWYG interfaces to work on each page and add graphics and interactivity, without having to code.

A lot of the design industry is aimed at marketing and eCommerce. If you're primarily developing services which people can access through Google and which involve browsing and buying, you're likely to use a no-code solution.

Designing with code

The part of the tech industry where I've carved out my career isn't generally about pubic-facing websites or eCommerce. Instead, I've made a living from designing software and complex interactions like password management and forms. I've spent a lot of time in the public sector where the average interaction is between a citizen and the government. In this domain, I've learned to use code-based tools to design tools.

Here's why: if you're writing content for interactive screens, you're no longer working on a 'document' in the traditional sense. You're creating a way to interact with a system. The interaction has to be easy to learn. You have to be able to offer a similar experience on laptops and smartphones. You have to cater for users who use a screen reader or who adjust the size of text on their device.

People who work in software, in apps and in online services therefore need to understand accessibility. Good content designers need a working knowledge of how to write HTML so that they can use the right conventions when they're writing content. They need to be close to the code.

Take the case of radio buttons. A set of radio buttons needs a legend at the top (usually a question) then a label for each button. It's not that easy to convey this on a Word document. It's easier to create the component on a real HTML page.

Who is the chosen one?

We rely on conventions to design online forms. Over the years the tech community has evolved ways of offering these components. For reference, try the W3C schools website or World Wide web consortium.

The HTML5 Editor and Codepen

It's possible to write your content in Word then hand it to an interaction designer or developer to turn into screens. However, I'd suggest you'll want to view your content on an actual screen to be able to tweak the words and get the tone of voice you want.

I often use the HTML5 editor to create and edit components. You can also use it if you're copying and pasting text from Word or Google docs. The most useful function is often the 'clear formatting' option which removes any unnecessary code.

The HTML5 editor doesn't allow you to save your content. To do this you can use a website like https://codepen.io/ if you're working on self-contained pages.

Serious code editors: Atom, VS Code, Sublime

If you need to be close to the code for the project you're working on, you may want to install one of the code editors used by developers.

It takes a bit of time to get used to this kind of software. If you're working with a designer or a developer with good code skills they'll help you get up to speed.

One of the big advantages you get from working this way is that you get access to add-ons which can help with your workflow. The 'Go live' function in VS code gives you a server to run a website locally on your machine. You can then view your web pages in your browser as you work.

Government design in the UK

The public sector in the UK has offered a common design system for the past decade. It's also created a prototyping kit for designers to build and host their own demo versions of online services.

The advantage of this is that designers are working with real HTML to create their web pages. If they work solely in Word and Google docs they're likely to have the wrong mindset. A Word document isn't a web page. Making prototypes in code you're working closely to the medium of the web.

Glitch websites for rapid content iterations

One of my favourite online services is provided by Glitch. They offer the ability to create free web projects with a Glitch domain. If you can use CSS then you can create realistic versions of web pages - see my Government data dashboard.

Now prototype it

This start-up provides a way for content editors to create and edit pages without the need for an external editor: Now prototype it.

For some designers this could be a useful solution as they can create prototypes and host them in the cloud. You can then share them with colleagues or run user research to test out your new website or service.

The future: designing with data

This topic deserves a whole blog post in itself, but briefly: anyone who uses an online service leaves a trail of data behind. This includes:

  • analytics data - what pages they visited and what parts of the service they used
  • user generated content - for example, messages and uploads
  • personal data - including their age, location and device

With data mining and machine learning, online services have more information than ever on the people who use a service. They can use this to improve the online service as well as offering analytical tools to their customers.

For content designers, understanding data is likely to be a useful skill in the future. Established industry tools include Tableau and Power Bi for data visualisation. You can also experiment with a free account at Observable HQ to try out their tools.

In my experience, these systems aren't easy to learn as they need you to have a working knowledge of statistics and well as learning some fairly demanding code. On the other hand, understanding this field might help you land a role where you work with data scientists and developers. The higher bar for entry into the data field means that the jobs are likely to be more secure and better-paid over time.

Conclusion

This has been a quick tour of the tools available to content designers. As Machine Learning changes the way we interact with computers it's likely these tools will evolve. At the same time, conventions like HTML and CSS are likely to remain the common language of the web. Content designers who understand code will able to to create accessible designs using conventions built up over the decades. That's likely to remain an advantage in a competitive jobs market.
